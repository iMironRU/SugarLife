# Концепция SugarLife / СладкаяЖизнь

> Рабочий документ. Версия 0.7.
> Имя проекта: SugarLife (англ.) / СладкаяЖизнь (рус.) — одно имя в двух языках.

---

## 1. Что это

Единое мобильное приложение для людей с диабетом 1 типа. Объединяет в одном опыте мониторинг глюкозы, ведение терапии, расчёт болюсов, обучение, статистику и автоматическое управление инсулинотерапией.

Заменяет связку из 4–5 разных приложений, которыми вынужденно пользуется современный диабетик: одно для мониторинга, другое для помпы, третье для статистики, четвёртое для CGM. И делает то, что эта связка делает плохо: настройку оборудования, переход от обучения к реальности, понимание происходящего без необходимости быть инженером.

Проект академический и сообщественный. Клиническая валидация — через партнёрство с медицинским вузом, разработка — через open-source сообщество.

---

## 2. Чем не является

- **Не платформа с плагинами.** Это не «ядро + сторонние приложения для статистики, графиков, авто-мониторинга». Прыжки между приложениями — корневая UX-проблема, которую проект решает, а не воспроизводит.
- **Не клон AndroidAPS / Loop с лучшей вёрсткой.** Архитектура переосмысляется с пользователя, не с алгоритмов.
- **Не обёртка вокруг Nightscout.** Nightscout — один из источников данных, не сердце системы.
- **Не игра с медицинскими функциями.** Игровой слой — это стратегия онбординга, не основной продукт.
- **Не коммерческое медицинское изделие на старте.** Академический исследовательский проект. Коммерциализация — отдельный вопрос на будущее.

## 3. Принципы

### 3.1 Пользовательский опыт

- **Всё в одном.** Один экран на каждую задачу. Никаких прыжков между приложениями за статистикой, графиком или настройкой устройства.
- **Action vs Analysis.** Экран действия («что делать сейчас») и экран анализа («что было») — это разные экраны с разными приоритетами. Сегодняшние приложения смешивают и проигрывают в обоих режимах.
- **Один аватар через все режимы.** CrankyPanky — гид от первой установки до экспертного использования. В Full Game — главный персонаж. В Light Mode — помощник. В Off — нейтральный аватар.
- **Игра как мост в реальность.** Диабетогочи — это не отдельный продукт. Это первый день жизни с приложением. Освоил основы в симуляторе → подключил настоящее железо → перешёл в боевой режим.
- **Прогрессивное раскрытие.** Функции открываются по триггерам готовности (освоил предыдущее → предложили следующее), а не по таймеру.

### 3.2 Железо (HAL)

- **Универсальный мастер подключения через камеру.** Фотография помпы → распознавание модели и серийного номера. То же для CGM, передатчиков, флаконов инсулина. Пользователь не должен переписывать цифры с маленькой наклейки.
- **Скрытая сложность.** Прошивки передатчиков, протоколы связи, методы шифрования — работа приложения, не пользователя. Если для работы с помпой нужно обновить прошивку — приложение делает это в фоне.
- **Платформенная консистентность.** iOS и Android должны вести себя одинаково для критических функций. «На iOS работает, на Android нет» — это не разница платформ, это баг.
- **Один код ядра для обеих платформ.** HAL, Data Core, Decision Engine, Watchdog — общая кодовая база для iOS и Android (KMM, Rust core, C++ core — конкретный стек определяется позже). UI может различаться платформенно, но логика — нет. Это единственный способ удержать консистентность на масштабе.
- **Цена ошибки настройки стремится к нулю.** Для каждого физического шага (вставить сенсор, нажать кнопку на передатчике) — визуальная инструкция «вот как это должно выглядеть», подтверждение перед следующим шагом. Сенсор стоит денег, убивать его из-за непонятной документации — провал системы, не пользователя.

### 3.3 Данные

- **Один источник правды.** IOB, COB, история — считаются в одном месте, показываются согласованно везде. Не «Nightscout говорит одно, локальный калькулятор другое».
- **Нулевой ручной ввод того, что система знает.** Заправил инсулин — приложение видит. Сменил сенсор — приложение видит. Замена инфузионного набора — приложение видит. Пользователь не должен дублировать в журнал то, что устройство уже сообщило.
- **Локально в MVP.** Без облака, без аккаунтов, без серверов. Данные пользователя — на его телефоне. Это сильно упрощает GDPR, безопасность и архитектуру.
- **Мульти-профили в архитектуре с первого дня.** Даже если в MVP активен один профиль — структура данных, изоляция и переключатель готовы. Иначе приложение бесполезно для родителя ребёнка с T1D с первого дня v2.
- **Проактивные вопросы вместо ретроактивных правок.** Если система видит аномалию (скачок сахара без записанного болюса, провал без записанной еды) — задаёт вопрос здесь и сейчас, не предлагает редактировать историю потом. Это снимает целый класс проблем с пересчётом IOB/COB и делает CrankyPanky агентом качества данных, а не просто гидом. Геолокация для предсказания вопросов («ты в кафе → собираешься есть?») — опционально, с явным согласием.

### 3.4 Безопасность

- **Watchdog между алгоритмом и помпой.** Независимый слой, проверяющий каждое предложение алгоритма на соответствие медицинским лимитам терапевтического профиля. Имеет право блокировать команду и переводить систему в безопасное состояние.
- **NO DATA как явный режим.** При разрыве связи дольше N минут экран меняется визуально: серый круг, явный таймстамп последнего значения, прогноза нет. Не «зелёный круг потому что последнее значение было нормальным» — это ложное чувство защищённости.
- **Свежесть важнее цвета.** Цветовой статус сахара отражает не только последнее значение, но и доверие к нему. Устаревшие данные — серый, не зелёный.
- **Симулятор визуально отличается от реальности.** Водяной знак, изменённый цвет фона, явный лейбл. Алармы и звуки в симуляторе отличаются от реальных. Иначе сонный родитель может перепутать.
- **Критические функции работают одинаково везде.** Пауза помпы, отмена временного базала, аварийная остановка петли — это не «опциональные фичи на одной из платформ».

### 3.5 Алгоритмы

- **Pluggable engine.** Петля — это заменяемый модуль. Можно взять oref0, oref1, написать свой PID, попробовать ML-модель. Архитектура не привязана к конкретному алгоритму.
- **In-silico перед in-vivo.** Каждый алгоритм проходит симуляцию на синтетических данных и записанных историях реальных пациентов перед тем как применять к живому человеку.
- **Pure functions.** Алгоритм получает состояние, возвращает предложение. Не имеет прямого доступа к помпе. Это даёт автоматизированное тестирование на миллионах сценариев.
- **Клиническая валидация через медвуз.** Без академического прикрытия алгоритмов — это идеи. С прикрытием — публикации, этическая комиссия, разрешение на клиническое исследование.
- **Каждое предложение алгоритма — в журнал.** С обоснованием: на каких данных основано, что предложил, что сделал Watchdog, что произошло дальше. Разбор полётов — основа доверия к системе.

## 4. Архитектура слоёв (концептуально)

На концептуальном уровне система делится на семь слоёв. Каждый слой имеет чёткую зону ответственности и общается с соседями через определённые контракты, а не через прямой доступ к внутренностям.

**HAL (Hardware Abstraction Layer).** Драйверы помп, CGM, передатчиков. Под капотом — конкретные протоколы (Bluetooth LE, NFC, проприетарные радиочастоты). В ядро отдаёт стандартизированные события и принимает стандартизированные команды. Сюда же — мастер подключения через камеру и обновление прошивок.

**Data Core.** Единое хранилище состояния. События (измерения, болюсы, еда, активность), производные (IOB, COB), история. Источник правды для всех остальных слоёв.

**Decision Engine.** Алгоритмы петли. Получают состояние из Data Core, выдают предложения изменить терапию. Не имеют прямого доступа к HAL.

**Safety Sandbox (Watchdog).** Независимый страж между Decision Engine и HAL. Проверяет каждое предложение на соответствие лимитам терапевтического профиля. Имеет приоритет над любым алгоритмом и право переводить систему в безопасное состояние.

**Game Layer.** Диабетогочи, CrankyPanky, обучающие уроки, симулятор. Работает поверх Data Core. Может использовать симулированные данные вместо реальных. Отдельная физиологическая модель для симуляции.

**UI Layer.** Единый интерфейс. Разделение action mode / analysis mode. Один аватар через все режимы. Главный экран — глюкоза в центре, периферия по бокам, точки входа в сценарии под ней, аналитика внизу.

**Integrations.** Nightscout (импорт и экспорт), Open Food Facts (база продуктов), потенциально Apple Health / Google Fit для синхронизации с экосистемой пользователя.

## 5. Пользовательский путь

**День 0: установка и онбординг.** Выбор роли — пациент, родитель, врач, любопытство. На основе роли — стартовый уровень игрового слоя. Пациент без опыта → Full Game. Опытный диабетик → fast path с тестом-проверкой. Родитель → Full Game с акцентом на симулятор и режим наблюдения.

**Дни 1–N: Full Game Mode.** Игра, обучение, симулятор. CrankyPanky ведёт уроки. Базовая программа: что такое сахар и его диапазоны, влияние еды (быстрые vs медленные углеводы), действие инсулина, гипо и реакция, гипер. Питомец Диабетогочи реагирует на действия в симуляции с физиологически правдоподобными задержками.

**Подключение железа.** Когда CrankyPanky видит, что пользователь готов к настоящим данным, предлагает подключить устройство. Фото помпы → распознавание → мастер подключения с пошаговой инструкцией → проверка прошивок и связи → калибровка.

**Переход в Light Mode.** Медицинский главный экран становится основным. Игра уходит в фон: Диабетогочи доступен из меню как отдельный режим «Уход за питомцем» (живёт независимо от гликемии — чтобы не давить на пользователя при гипо). CrankyPanky остаётся как гид по приложению.

**Включение петли (опционально, с явным подтверждением).** Алгоритм работает, Watchdog защищает, пользователь видит каждое предложение и решение в журнале. Включение требует выверенного терапевтического профиля.

**Эксперт.** Всё на главном экране, игровой слой выключен, CrankyPanky минимально присутствует. Доступ ко всем функциям, продвинутые настройки, экспорт для эндокринолога.

**Edge cases.** NO DATA при разрыве связи (явный режим). Отзыв временных команд при потере связи с помпой. Конфликт источников — глюкометр приоритетнее CGM при калибровке. Визуальная и звуковая дифференциация симулятора и реальности.

## 6. Что было решено по дороге

### Продуктовые

- Имя проекта: **SugarLife / СладкаяЖизнь** (англ. и рус. версии одного имени)
- Не моно-плагин-платформа, а единое приложение
- Включает автоматическое управление, не только мониторинг
- Медвуз — полноценный участник проекта, не просто консультант. Имеет лицензию на сертификацию алгоритмов в РФ. Это даёт путь к легальному медицинскому применению, а не только к open-source-эксперименту.

### Архитектурные

- IOB/COB — считаем сами, не зависим от Nightscout
- Болюсный калькулятор — нет отдельного экрана. Есть «объявление еды», который при включённой петле показывает действия петли, при выключенной — рассчитанную рекомендацию
- Редактирование задним числом архитектурно исключаем — заменяем проактивными вопросами в момент аномалии
- Backend на v3 — собственный, но Nightscout-совместимый по API (для миграции существующих пользователей и работы со сторонними клиентами)
- Один код ядра на iOS и Android. UI может различаться, ядро (HAL, Data Core, Decision Engine, Watchdog) — нет. Конкретный стек выбираем позже.
- Алгоритм петли — собственный, не «обёртка над oref». Опирается на существующие наработки, но кастомизируется под нашу модель данных и проактивную архитектуру. Прорабатывается отдельной веткой работы.

### Игровые / UX

- ХЕ не считаем, считаем углеводы в граммах
- База продуктов — помощник, нам достаточно количества углеводов (БЖУ не нужен)
- CrankyPanky как имя оставляем, образ уже есть. В Light Mode — агент качества данных, не только гид.
- Три уровня игрового слоя: Full Game / Light / Off
- Уход за питомцем отделён от медицинских метрик
- Мульти-профили в архитектуре с первого дня, активны со v2
- Локально в MVP, облако в v3

### Методология

- UX-прототипы делаем в HTML/web прежде чем выбирать нативный стек. Это даёт три выгоды: ранняя валидация интерфейсов, лёгкое тестирование сообществом (открыл в браузере), отложенный выбор технологий имплементации до момента ясности.

## 7. Открытые вопросы

### Параллельная ветка работы (не блокирует UX)

- **Алгоритм петли.** Обзор существующих (oref0, oref1, OpenAPS, CamAPS, академические PID и MPC), выбор базы для нашей реализации, кастомизация под нашу модель данных и проактивную архитектуру, план клинической валидации с медвузом. Это самостоятельный документ.

### Архитектурные (для UX-прототипа не критичны, нужны до нативной имплементации)

- Технологический стек ядра — KMM, Rust, C++? Решаем после первых HTML-прототипов.
- Database на устройстве — SQLite, Realm, что-то ещё?
- Какие пороги «тревожного прогноза» — формула на основе горизонта и направления?
- Как именно мы детектируем аномалии для проактивных вопросов (порог отклонения от прогноза, временное окно)?

### UX (нужны для прототипов)

- Когда переход из Full Game в Light Mode — тест на знания, время, выбор?
- Какие конкретные действия в «уходе за питомцем» в Light Mode?
- Как опытному пользователю пропустить Full Game — кнопка «пропустить» или тест-проверка?
- Настройки навязчивости CrankyPanky — сколько уровней?
- Сколько обучающих уроков нужно для перехода в Light Mode — пять базовых хватит?
- Источник базы продуктов в России — Open Food Facts покрывает плохо, какая альтернатива?

### Игровой слой

- Симулятор и реальный режим — отдельные пространства или общий журнал с фильтром?
- Питомец — один на пользователя или по одному на подопечного (для родителя)?

### Privacy

- Геолокация для проактивных вопросов — как реализовать с минимальным privacy-следом? Опционально с явным согласием, но какой именно сигнал брать (геозоны? координаты? Wi-Fi-точки?)
